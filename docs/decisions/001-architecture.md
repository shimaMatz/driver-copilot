# ADR-001: アーキテクチャ基本選定

- **ステータス**: 承認済み
- **日付**: 2026-04-06
- **決定者**: 開発チーム

## コンテキスト

長距離トラックドライバー向けのナビゲーション支援アプリを新規開発する。iOS / Android の両プラットフォームに対応し、地図表示・音声操作・運転時間管理を主要機能とする。少人数チームで迅速にプロトタイプを構築し、段階的に機能を拡充する方針である。

## 決定事項

### D-1: Expo SDK 54 の採用

**選択**: Expo SDK 54 (React Native 0.81.5, React 19.1.0)

**理由**:
- iOS / Android のクロスプラットフォーム開発を単一コードベースで実現できる。
- Expo Go による即時プレビューで開発サイクルが高速。
- expo-location, expo-speech など、本アプリに必要な機能のマネージドモジュールが揃っている。
- OTA アップデートにより、ストア審査を経ずに軽微な修正をデプロイ可能。

**トレードオフ**:
- Expo Go では `@react-native-voice/voice` が動作しないため、音声認識機能のテストには開発ビルドが必要。
- ネイティブモジュールの選択肢が Expo 対応のものに限定される。

**却下した選択肢**:
- **Flutter**: チームの React / TypeScript 経験を活かせない。
- **素の React Native**: ネイティブモジュール管理やビルド設定の負担が大きい。
- **ネイティブ (Swift / Kotlin)**: 2プラットフォーム分の開発コストがかかる。

### D-2: Feature-based ディレクトリ構成

**選択**: `src/features/` 配下に機能単位でディレクトリを分割する構成。

```
features/
  navigation/   # ナビゲーション（画面・API・フック・型）
  driving-timer/ # 運転時間管理
  voice/         # 音声入出力
  demo/          # デモモード
```

**理由**:
- 機能の凝集度が高く、関連するコンポーネント・フック・型・APIクライアントが同一ディレクトリに収まる。
- 新機能の追加時に既存ディレクトリへの影響が限定的。
- 機能単位でのコードオーナーシップが明確になる。

**トレードオフ**:
- 機能間で共有するコードは `shared/` に配置する規約が必要。
- 機能が増えるとディレクトリが多くなるが、各ディレクトリの責務は明確に保たれる。

**却下した選択肢**:
- **レイヤーベース** (`components/`, `hooks/`, `services/`): 関連するコードが分散し、機能の全体像を把握しにくい。

### D-3: Google Maps API + オープンソースフォールバック戦略

**選択**: Google Maps Platform を主たる地図・ルート・場所検索 API とし、OSRM / Nominatim / Overpass API をフォールバックとして併用する。

| 機能 | プライマリ | フォールバック |
|------|-----------|--------------|
| ルート検索 | Google Directions API | OSRM |
| 場所検索 | Google Places Autocomplete | Nominatim |
| POI検索 | Google Places Nearby | Overpass API |

**理由**:
- Google Maps Platform は日本国内のルート品質・POI カバレッジが最も高い。
- 有料道路/一般道の切替、IC/JCT/SA 名称の取得が可能。
- フォールバックにより、APIキー未取得の開発初期やコスト超過時にも基本機能が維持できる。
- デモモードと合わせて、ゼロコストでアプリの価値検証が可能。

**トレードオフ**:
- Google Maps Platform は従量課金（月 $200 無料枠あり）。利用量が増えるとコスト管理が必要。
- OSRM フォールバックでは有料道路/一般道の区別や詳細なステップ情報が得られない。
- Nominatim / Overpass はレート制限が厳しく、大量ユーザー対応には不向き。

**却下した選択肢**:
- **Google Maps のみ**: APIキー未設定時に全く動作しなくなる。
- **OSM のみ**: 日本国内の商用 POI データが不足。有料道路判定が困難。
- **MapBox**: Google と同等の有料サービスだが、日本国内のデータ品質で Google に劣る。

### D-4: JR東日本アプリ風デザインシステム

**選択**: JR東日本アプリの公式サイトから抽出したカラーパレット・タイポグラフィを基調としたデザインシステム（DESIGN.md に定義）。

- プライマリカラー: `#35A86E` (JR Green)
- テキストカラー: `#34525F` (Deep Navy)
- 背景: `#FFFFFF` (白)
- フォント: Noto Sans JP

**理由**:
- 鉄道・交通系アプリとしての信頼感と清潔感を表現できる。
- 緑のアクセントカラーは安全・安心のイメージと合致し、ドライバー向けアプリに適している。
- 白背景 + 濃い文字色の組み合わせは可読性が高く、運転前後の確認に適している。
- 既存の成功事例（JR東日本アプリ）をベースにすることで、デザイン品質を効率的に確保できる。

**トレードオフ**:
- JR東日本の公式ガイドラインではないため、細部の解釈に揺れが出る可能性がある。
- モバイルネイティブ（React Native）への適用時に、Web ベースの vw 指定をネイティブの dp/sp に読み替える必要がある。

## 影響

- フロントエンドは Expo + TypeScript で統一し、ネイティブコード記述は原則不要。
- バックエンドは REST API（`/api/v1/`）で疎結合にし、フロントエンドとは独立してデプロイ可能。
- 外部API依存はフォールバック戦略により段階的に導入可能。デモモードでゼロ依存の動作を保証。
